MLOps Principles

MLOps establishes best practices for testing, deploying, managing, and monitoring machine learning models in production environments. The goal is to eliminate "technical debt" in ML applications by treating ML assets consistently with other software assets in CI/CD environments.

THREE PHASES OF MLOPS

Phase 1 - Designing the ML-powered Application:
Focuses on business understanding, data understanding, and solution design. Teams identify use cases, assess project viability, examine available data, and specify functional and non-functional requirements.

Phase 2 - ML Experimentation and Development:
Validates ML applicability through proof-of-concept implementations. Teams iteratively identify suitable algorithms, perform data engineering, and build models with stable quality suitable for production deployment.

Phase 3 - ML Operations:
Focuses on production delivery using established DevOps practices including testing, versioning, continuous delivery, and monitoring.

THREE LEVELS OF ML AUTOMATION MATURITY

Level 0 - Manual Process:
Data scientists execute every step manually, typically using Jupyter Notebooks. This experimental, iterative approach is typical for initial ML implementations. Common at early-stage startups or for first-time ML projects.

Level 1 - ML Pipeline Automation:
Model training executes automatically when new data arrives. Includes continuous model retraining and data/model validation steps. The training pipeline is automated; deployment may still be manual.

Level 2 - CI/CD Pipeline Automation:
Automated systems build, test, and deploy data, models, and training pipelines. This represents the highest automation maturity. Rare even at large tech companies.

MLOps SETUP COMPONENTS (for Level 1+)

- Source Control: versioning code, data, and ML artifacts (Git, DVC)
- Test & Build Services: quality assurance through CI tools (Jenkins, GitHub Actions)
- Deployment Services: pipeline deployment via CD tools
- Model Registry: storage for trained models with metadata (MLflow Registry)
- Feature Store: preprocessing and feature management (Feast, Tecton)
- ML Metadata Store: tracking model training information (ML Metadata)
- ML Pipeline Orchestrator: automating experiment steps (Kubeflow, Airflow)

CONTINUOUS X PRACTICES IN MLOPS

Continuous Integration (CI): Extends traditional code testing to include testing and validating data and models. Not just "does the code run?" but "is the data schema correct?" and "does the model meet minimum performance thresholds?"

Continuous Delivery (CD): Automatically deploys ML training pipelines and prediction services.

Continuous Training (CT): Unique to ML systems. Automatically retrains models for redeployment when data distribution shifts or model performance degrades.

Continuous Monitoring (CM): Tracks production data and model performance metrics tied to business objectives.

VERSIONING IN ML SYSTEMS

Why versioning is harder in ML than in software:
- ML models change due to: new data, new training methodologies, performance degradation over time, deployment in new applications, regulatory compliance requirements
- You need to version three things simultaneously: code, data, and model weights
- Reproducing a past model means rerunning with the same code, same data, and same random seeds

Every ML model specification should undergo code review and version control for auditability and reproducibility.

Tools: DVC (extends Git for data versioning), MLflow (experiment tracking + model registry), Weights & Biases (experiment tracking + visualization).

TESTING IN ML SYSTEMS

Complete ML testing encompasses three scopes:

Features and Data Tests:
- Data validation checking schema and domain compliance (column types, value ranges)
- Feature importance testing - do your features actually have predictive power?
- Privacy/policy compliance (GDPR compliance, no PII leakage)
- Unit tests for feature creation code

Model Development Tests:
- Algorithm verification: does your model align with business objectives?
- Model staleness testing: when was the model last retrained?
- Comparison with simple baseline models (sometimes a logistic regression beats your deep net)
- Fairness and bias assessment across demographic groups
- Unit testing for model code

ML Infrastructure Tests:
- Reproducibility verification under identical conditions
- API stress testing and crash recovery
- Full pipeline integration testing
- Pre-deployment model validation (offline evaluation before going live)
- Canary deployment testing (shadow mode, send real traffic to new model but don't act on its predictions)
- Training/serving feature parity (train-serve skew is a leading cause of production failures)

ML TEST SCORE (Production Readiness)

Score interpretation:
- 0: Research project, not productionized
- (0, 1]: Untested, potential reliability issues
- (1, 2]: Initial productionization attempted
- (2, 3]: Reasonably tested, more automation possible
- (3, 5]: Strong automated testing and monitoring
- > 5: Exceptional testing and monitoring

MONITORING

Production monitoring requires continuous oversight:
- Dependency changes (new package versions that could change model behavior)
- Data schema validation (is incoming data still the right format?)
- Consistent feature computation between training and serving environments
- Numerical stability (NaN/infinity detection - these silently corrupt predictions)
- Computational performance (latency, throughput, memory)
- Model staleness measurement (time since last retraining)
- Prediction quality degradation detection (precision, recall, F1 decline)

Model decay triggers: concept drift (relationship between features and labels changes), data drift (input distribution changes), upstream data pipeline failures.

REPRODUCIBILITY

Each phase must produce identical results from identical inputs:

Data Collection: backup datasets, create snapshots, timestamp data sources, implement versioning.

Feature Engineering: version feature code, ensure data collection reproducibility.

Model Training: ensure consistent feature ordering (shuffling order matters!), document hyperparameter selection.

Deployment: match software versions across environments, use containerization (Docker), maintain consistent programming language and library versions.

TRAIN-SERVE SKEW: THE SILENT KILLER

The most common production ML failure mode: the model gets different features at serving time than it saw during training. Causes:
- Different code paths for batch (training) vs real-time (serving) feature computation
- Training uses perfect historical data; serving uses imperfect real-time data
- Different data preprocessing in training pipeline vs serving pipeline

Prevention: use a shared feature store that serves both training and serving from the same code path.

KEY TAKEAWAY FOR ML PRACTITIONERS

The gap between a 95% accuracy model in a notebook and a reliable production ML system is enormous. Production ML requires:
- Automated retraining pipelines
- Data and model versioning
- Comprehensive testing at data, model, and infrastructure levels
- Monitoring for drift and degradation
- Reproducible experiments
- Clear deployment and rollback procedures

Strong engineering foundations are essential. The best ML engineers combine solid software engineering practices with ML expertise.
